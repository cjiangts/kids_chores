# Type IV Backend Design

## Goal

Add a new `type_iv` deck category for dynamically generated practice, initially powered only by Python snippets and graded on the backend.

The first target use case is math practice such as `?? + ??`.

## Core Product Behavior

- Kid sees one generated prompt at a time.
- Kid types an answer and taps `Done`.
- The app moves on immediately without revealing correctness.
- At the end, the app shows summary counts only.
- Backend does grading.
- Python snippets may optionally provide distractor answers for future multiple-choice UI.

## Key Design Decisions

- `sessions` stays category-based, not deck-based.
- Kid `cards` table remains static and should not store per-session generated problems.
- Each opted-in Type IV deck has exactly one representative card in the kid DB.
- That representative card stands for the deck pattern, not an individual generated problem.
- Aggregated stats for Type IV are intentionally pattern-level, not per generated problem.
- The exact generated prompt/answer payload is stored in a Type IV sidecar table keyed by `session_results.result_id`.
- Retry answers are stored in `submitted_answers` as an array on that sidecar row.
- No new foreign key constraints are added.
- Type IV shared deck definitions are immutable after creation.
- Only Python is supported; no language field is needed.

## Shared DB Changes

File: `backend/src/db/shared_deck_schema.sql`

### `deck_category`

- Start allowing `behavior_type = 'type_iv'` in backend logic.

### `deck`

- No new `deck_kind` column.
- No `display_name` column.
- The friendly label can live in the special representative card's `front`.

### New table

```sql
CREATE TABLE IF NOT EXISTS deck_generator_definition (
  deck_id INTEGER PRIMARY KEY,
  code TEXT NOT NULL,
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

Rules:

- If a shared deck belongs to a `type_iv` category, it must have one `deck_generator_definition` row.
- Type IV shared decks do not use normal shared generated-question card rows.
- A single representative shared card row may still be materialized for label/display purposes.

## Kid DB Changes

File: `backend/src/db/schema.sql`

No new Type IV session table is needed.

No per-session generated rows should be inserted into kid `cards`.

### New table

```sql
CREATE TABLE IF NOT EXISTS type4_result_item (
  result_id INTEGER PRIMARY KEY,
  prompt VARCHAR NOT NULL,
  answer VARCHAR NOT NULL,
  distractor_answers VARCHAR[] NOT NULL DEFAULT [],
  submitted_answers VARCHAR[] NOT NULL DEFAULT []
);
```

Meaning:

- `result_id` points to one existing `session_results.id`
- `prompt` is the generated question shown to the kid
- `answer` is the canonical server-side correct answer
- `distractor_answers` stores optional wrong choices generated by Python
- `submitted_answers` stores the first typed answer plus any later redo answers

## Representative Card Model

When a kid opts into a Type IV shared deck:

- create one kid-local `decks` row for that deck
- create one and only one kid-local `cards` row for that deck

That single card is the representative card for the deck.

For Type IV:

- `session_results.card_id` points to the representative card
- many different generated problems from the same deck may share that same `card_id`
- this is intentional because Type IV stats are deck-pattern stats

## Session Flow

### Start session

- Resolve the category as usual.
- Load all opted-in Type IV decks in that category.
- For each generated prompt chosen for the session:
  - run the deck's Python generator
  - keep `prompt`, `answer`, and optional `distractors` in pending-session memory only
- Send only prompt payload to the frontend; do not send answers.

### Complete session

For each answered generated problem:

- insert one normal `session_results` row
  - `session_id`
  - representative `card_id`
  - `correct`
  - `response_time_ms`
- insert one `type4_result_item` row using that `result_id`
  - `prompt`
  - `answer`
  - `distractor_answers`
  - `submitted_answers` with the first typed answer

Unanswered generated prompts are discarded and are not persisted.

## Retry / Redo Flow

- Retry logic for Type IV cannot rely on distinct `card_id`, because all generated problems from one deck share the representative card.
- Type IV retry logic should instead operate on `type4_result_item` joined through `session_results`.
- On redo of a previously wrong generated problem:
  - do not create a new generated-problem row
  - append the new typed answer to `submitted_answers`
  - update retry/session aggregate fields using the existing retry model

## Python Generator Contract

Initial shape:

```python
def generate(rng):
    return {
        "prompt": "11 + 22",
        "answer": "33",
        "distractors": ["32", "34", "44"],
    }
```

Rules:

- `prompt` required
- `answer` required
- compare answers as exact canonical strings, aside from normal outer trimming
- `distractors` optional
- distractors must be non-empty strings, deduped, and must not include the correct answer

## Why This Design

- Keeps the existing static beauty of `cards`.
- Preserves the current category-based meaning of `sessions`.
- Avoids forcing `session_results.card_id` to become nullable.
- Gives Type IV a clean future path to multiple-choice UI.
- Supports parent review later with prompt, expected answer, and submitted answers.
- Keeps bank/card UI aligned with product intent: one Type IV deck appears as one card.
